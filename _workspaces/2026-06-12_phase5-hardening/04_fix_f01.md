## Problem (recap)

`credits_ledger` had no UNIQUE constraint on `ref`, and `creditsRepo.spend()` performed only a
conditional debit with NO ref-dedup (only `grant()` checked `hasLedgerRef`). Result: a re-run of an
already-charged spend (DLQ replay / manual requeue after a status reset / retried request) would
write a SECOND debit → double-charge, violating a financial invariant. `requireCredits` made it
worse by using `ref = tool` (a constant), so the ref was meaningless for the tool path.

## Fix order (deliberate — must NOT break legitimate repeated spends)

The key insight: the unique key must be **per logical operation, stable across retries** — NOT a
constant shared by distinct operations. So we changed the refs FIRST, then added the constraint.

1. **`requireCredits.ts`** — spend ref changed from the constant `tool` to a PER-REQUEST unique
   `spend:{tool}:{requestId}`. `requestId` comes from the logger middleware (`c.get('requestId')`,
   set as `REQUEST_ID_KEY` in `logger.ts`), falling back to `crypto.randomUUID()` if absent. Two
   *distinct* valid calls to the same tool now get two distinct refs (the unique index never blocks
   a legitimate second spend), but a *retry of the same request* reuses the ref → idempotent.

2. **`creditsService.spendCredits(userId, tool, ref?)`** — added an optional `ref` param
   (`services/types.ts` contract updated). The service passes `ref ?? tool` to the repo, so legacy
   2-arg callers keep the old behaviour and new callers supply an idempotency key.

3. **`consume.ts`** — UNCHANGED. It already spent with `ref: job:{jobId}` directly via
   `repo.spend()` (QA's per-job fix). That ref is now actually enforced by the constraint + the
   repo dedup, so a job replay is a true no-op.

   **`llm-tools.ts` (chat SSE) — REQUIRED extra fix (regression prevention).** The chat
   stream's post-`[DONE]` deduct used `spendCredits(user.id, CHAT_TOOL)` (constant ref). With the
   new unique index a user's SECOND chat would reuse `ref = CHAT_TOOL` → my idempotent spend would
   silently no-op → **under-charge**. Changed to `spend:{CHAT_TOOL}:{requestId}` (same per-request
   pattern as requireCredits). This was the ONLY other constant-ref spend caller; without it the
   unique index would have broken chat billing. consume.ts + requireCredits are the only remaining
   spend paths, both now using unique refs. (Grep `spendCredits|\.spend(` confirms no other caller.)

4. **`migrations/0005_ledger_idempotency.sql`** (NEW) — PARTIAL unique index
   `idx_ledger_ref_unique ON credits_ledger(user_id, ref) WHERE ref IS NOT NULL`.

5. **`creditsRepo.spend()`** — now idempotent (details below).

## creditsRepo.spend() — the atomic + idempotent design

Single `db.batch()` (one D1 transaction, statements run in order, each sees the prior's effect):

- **Stmt 1 (INSERT, the authoritative gate):** `INSERT … SELECT … FROM subscriptions WHERE
  user_id=? AND credits_balance >= cost AND NOT EXISTS(ledger row with this user_id+ref)`. It lands
  iff the user can afford it AND the ref isn't already charged. `balance_after` is computed from the
  PRE-debit cache. Success is read from THIS statement's `changes()`.
- **Stmt 2 (cache UPDATE, derived):** `credits_balance = (SELECT COALESCE(SUM(delta),0) FROM
  credits_ledger WHERE user_id=?)`. Re-derives the cache from the ledger (the source of truth,
  BR-007) instead of a blind `balance - cost`. This makes the cache update itself **idempotent** —
  re-running after a no-op INSERT recomputes the same value, so the cache can never drift or be
  double-debited. (An earlier draft used `balance = balance - cost` keyed to the just-inserted row;
  that had a cache-drift hole when a later grant restored the old balance — rejected.)
- **UNIQUE-violation catch:** if two concurrent transactions both pass `NOT EXISTS`, the second
  INSERT raises `UNIQUE constraint failed` and aborts its whole batch (no debit). `spend()` catches
  it (`isUniqueViolation()` matches the SQLite message), and for a non-null ref returns idempotent
  success with the current balance. This is the hard backstop the partial index guarantees.
- **Outcome decoding:** INSERT `changes()===1` → charged now (ok). Else recompute balance and: if a
  ledger row with this ref exists → already charged → idempotent `ok:true` no second debit; else →
  genuinely insufficient → `ok:false` (402). Null-ref spends behave exactly as before (`ref = ?`
  never matches NULL, so NOT EXISTS is always true and the index excludes them).

## Migration & OLD-DATA handling

- **Partial index (`WHERE ref IS NOT NULL`)** was chosen because **signup grants are written with
  `ref = NULL`** (the auth hook calls `grantPlanCredits(user,'free')` with no ref) and must NOT
  collide. Grant invoice refs (`invoice:{id}`) are already unique-by-construction (grant() dedups via
  `hasLedgerRef`, one invoice per cycle), so they coexist fine with the index.
- **Pre-flight cleanup of legacy constant-ref spend rows:** before creating the index, the migration
  runs one `UPDATE` that nulls out the ref of *duplicate* legacy spend rows (negative delta, ref =
  an old constant tool name, NOT the new `spend:%:%`/`job:%` formats), keeping the earliest row per
  `(user_id, ref)`. This prevents index creation from failing on pre-existing dev data where a user
  spent the same tool twice under the old constant ref. On a fresh D1 it matches 0 rows (no-op).
  We are pre-prod (dev D1 only), so this one-time normalization is safe and documented here.
- **Verified on real D1** (`wrangler d1 migrations apply herorank --local` on a fresh DB):
  - All 5 migrations apply ✅ (0005 = ✅).
  - Duplicate non-null spend ref → `UNIQUE constraint failed: credits_ledger.user_id, credits_ledger.ref` ✅ (this is the exact text `isUniqueViolation()` matches).
  - Two `ref = NULL` grant rows for the same user coexist ✅ (partial index excludes NULL).

## Double-charge: CONFIRMED BLOCKED

- **Job replay (consume.ts):** `ref = job:{jobId}` is now enforced. A re-run of an already-charged
  job (status reset → requeue, DLQ replay) re-enters `repo.spend()` with the same ref → INSERT
  blocked by `NOT EXISTS` / unique index → idempotent no-op → NO second debit.
- **Request retry (requireCredits):** `ref = spend:{tool}:{requestId}` — a retried request with the
  same requestId is idempotent; two genuinely distinct calls get distinct refs and both charge.
- **Concurrent same-ref:** unique index aborts the loser's batch → exactly one debit.
- The non-double-charge property now holds at the **ledger/storage layer**, independent of the job
  lifecycle (which remains as a second line of defence).
